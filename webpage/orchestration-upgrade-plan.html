1 · TL;DR & what changed today

The repo's documentation described a system that was never deployed. Per-project graph isolation, the relational-edge model, and the self-healing seeder were all committed to git but had never run. The live server had one graph and a schema that rejected the edges the skills mandate — so every agent obeying structured-memory/SKILL.md was silently failing.

The deeper pattern: this repo has no mechanism that verifies its own claims. Documentation, seeds, compose files, and scripts drifted apart because nothing executes them. Adding more skills won't help until the loop closes — which is what §6 is about.

The single highest-leverage change is not more agents. It is an intent/spec loop in front of the orchestrator (§6, L0) plus gating Best-of-N on the existence of an objective arbiter (§7). Today the risk model treats ambiguity as a reason to spawn more parallel engineers — which is precisely backwards.

3 · Diagnosis: three structural faults

A · Nothing verifies the claims

Docs, seeds, compose, and scripts describe behaviour that no test exercises. apply-cluster.sh was committed broken and stayed broken because running it was never part of any loop. Every bug in §2 is a drift bug, not a logic bug.

Fix: a verify-stack skill + CI-ish check that asserts live schema == memory.pg, graphs == cluster.yaml, seeds == live, and configs reference real networks.

B · Inverted information density

The entry point is 58 lines; the internals are ~1,300. swarm-orchestration describes itself four times (SKILL.md 287 + FRAMEWORK 254 + RATIONALE 431 + README 123 + VS_OPENHANDS 226), while the five vendored skills are 16–29-line stubs.

An agent reads the thin router, learns the wrong things (§2 #9), and never reaches the depth. Fix: thicken the router into a dispatch table; collapse the orchestration quartet to one skill + one rationale.

C · The loop never closes

The flow in swarm-orchestration/SKILL.md runs task → contract → engineer → gates → review → merge. It never writes back to memory, and nothing measures the run. So the next session re-derives the same context and repeats the same mistakes.

Fix: L4 (persist) + L5 (measure) in §6. Those two steps are what make it a loop rather than a pipeline.

4 · The router skill

Decision: extend skills/repository-index/SKILL.md. It already carries the router role and a frontmatter description that makes it auto-discoverable. SYNC.md becomes VENDORING.md.

The router's job is progressive disclosure: an agent reads ~100 lines and knows exactly which 2–3 files to open next — never the whole library. swarm-orchestration already states the rule ("Do not expose a large undifferentiated skill library to all agents"); the router is how that rule is enforced.

Required contents

  1. Fix the MCP table. Correct the superpowers row (#9). Add context7 (library docs — prefer over web search). Note Serena's excludeTools trap: memory tools are filtered in the self-hosted stdio config but enabled in some project configs, so write_memory may or may not exist.
  2. Add the 5 missing internal skills — structured-memory, coding-principles, html-working-documents, mcp-servers-setup, repository-index itself.
  3. Trigger column must be a predicate, not a vibe. "Any complex task" is unroutable. Use observable conditions: "diff > 3 files", "answer would exceed ~100 lines of markdown", "about a third-party library's API".
  4. Declare each entry's tier + dependencies (§5) so an agent on a laptop degrades gracefully instead of failing on a missing container.
  5. Route the loops (§6), not just the skills: which loop am I in, what closes it, what's the exit gate.
  6. Precedence. State the order when several apply: memory recall → process skill (brainstorming / systematic-debugging) → domain skill → verification. Today two systems (superpowers:using-superpowers and this router) both claim "check me first" with no tiebreak.

Anti-goal. Do not let the router become a catalog that restates each skill. It is a dispatch table: trigger → file → tier. Every line that duplicates a SKILL.md is a line that will drift (fault A). This is the repo's own Principle 1.

5 · Piecewise use / weak hardware

Today the stack is presented as all-or-nothing, yet most value needs no infrastructure at all. Declare four tiers; make every skill state its minimum tier and its degradation path.

TierNeedsGets youDegradation when absent
T0 — papernothing (files only) All skills, the router, contracts, checkpoints as plain markdown. coding-principles, TDD, html-working-documents, the loops in §6. —
T1 — laptop+ Serena (uvx, LSP) Semantic navigation/refactor. The single biggest token win. Fall back to Glob/Grep + targeted reads.
T2 — workstation+ Docker: Omnigraph + MinIO (~1.5 GB), Graphify Durable typed memory; code-structure graph. Memory → a committed MEMORY.md per repo; graph → Serena refs.
T3 — homelab+ viewer, Authelia, sync timers, observability, Herdr Multi-device, multi-user, remote, dashboards. Local main only; no sync.

Concrete changes

  • Add tier: and requires: to every skill's frontmatter. The router groups by tier.
  • T2 embeddings run on CPU (nomic-embed-text, 768-dim) — already true, but undocumented as a weak-hardware property. Say so; it's a selling point.
  • Ollama is not required for memory to work — only for nearest() ranking. Without it, recall degrades to graph traversal + scalar indexes, which is fine at this graph's size (162 nodes). State this; it removes the biggest perceived hardware barrier.
  • The mem0-fallback profile is dead weight now (Omnigraph is the decision, ADR-0001, and mem0 pulls Postgres + pgvector + a dashboard + an LLM key). Recommend deleting it — see §8.

Why this matters beyond weak hardware

Tiering is what makes the repo adoptable. Right now a newcomer must stand up MinIO to use a TDD skill. Tiers turn it into: copy one starter → get value at T0 → opt into infra when the pain justifies it.

It also bounds blast radius: a T0/T1 agent physically cannot corrupt a shared graph.

6 · The loop architecture

Grounded in Anthropic's five workflow patterns (chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer) and the 2026 harness-engineering layer model (tool orchestration · verification · context/memory · guardrails · observability). The repo has the first four in some form; it is missing intent at the front and observability at the back.

  ┌── L0 INTENT ────────┐   gate: every acceptance criterion is an executable command
  │ restate → find      │
  │ ambiguity → ask ≤3  │──► SPEC.md  (goals · non-goals · acceptance · verify cmds)
  │ → recall memory     │        │
  └─────────────────────┘        ▼
  ┌── L1 PLAN ──────────┐   gate: contract names files, cmds, forbidden changes
  │ decompose → contract│──► ARCHITECTURE_CONTRACT.md + risk score (on a CLARIFIED spec)
  └─────────────────────┘        │
                                 ▼
  ┌── L2 IMPLEMENT ─────┐   evaluator = THE TEST SUITE (cheap, objective, unbiased)
  │ red → green → refac │◄──┐  loop until green; Best-of-N only if tests arbitrate
  └─────────────────────┘   │
                            ▼
  ┌── L3 REVIEW ────────┐   parallel dimensions (independent errors) → triage → verdict
  │ swarm → adversarial │──► BLOCKED / REQUEST_CHANGES / APPROVE     (max 3 loops)
  │ verify → triage     │        │
  └─────────────────────┘        ▼
  ┌── L4 INTEGRATE ─────┐   ** closes the loop **
  │ gates → merge →     │──► persist Decision/Rule to Omnigraph + changelog/ADR + cleanup
  │ PERSIST MEMORY      │        │
  └─────────────────────┘        ▼
  ┌── L5 MEASURE ───────┐   tokens · wall-clock · rework count · verdict mix · escalations
  │ evals → tune        │──► feeds risk weights + routing back into L0/L1
  └─────────────────────┘
L0 · Intent loop missing todayexpert · single strong model

The repo's risk model gives ambiguous acceptance criteria its highest weight (+0.20) — and then responds to it by spawning more parallel engineers. That is backwards. Ambiguity is a bias problem (wrong target); ensembles only reduce variance. N engineers aimed at an unclear target miss together, in a correlated way, and you pay N× to discover it.

Do instead: ambiguity must be driven to zero before risk is scored. Ask at most three blocking questions, write the answers back into SPEC.md, and only then score risk. This mirrors Spec-Kit's /clarify, which GitHub credits with roughly an order of magnitude fewer regenerate-from-scratch cycles.

Exit gate (hard): every acceptance criterion names a command that passes or fails. If you cannot name the command, the spec is not done — and L2's evaluator would have nothing to run. This one gate is what makes L2 and §7 work.

L1 · Plan loopexpert · judge panel only if space is wide

Mostly exists. Change: risk is scored after L0, so the ambiguity signal is normally 0 and the score reflects genuine structural risk (blast radius, security, concurrency). Use a 3-way judge panel only when the design space is genuinely wide — competing approaches, not competing implementations.

L2 · Implement loopevaluator-optimizer · evaluator = tests

The highest-value loop, because the arbiter is free, objective and unbiased. Keep TDD as the inner loop; the verification commands come from L0's spec, not invented per task. no-mistakes (local proxy validation) is the right shape here — but it's a 16-line stub and needs to be real: exact commands, exit codes, what to do on red.

L3 · Review loopcrowd wins here

qa-swarm's persona split (Security / Performance / XP) is genuinely good: the dimensions decompose and their errors are independent — the textbook conditions for ensembling.

Add an adversarial verify stage. Each finding gets independently challenged before it reaches triage; drop it unless it survives. Research on verification dynamics shows verifier capability hits sharp diminishing returns — so buy diverse lenses, not more of the same reviewer. Prefer 3 different personas over 3 samples of one.

L4 · Integrate loop gapcloses the loop

The orchestration flow's 11 steps never persist memory. Add it as a required step, not an aspiration: on approval, write the Decision (+ rationale) and any new Rule to the project's Omnigraph graph with a hub edge and ≥1 relational edge, then verify with commits_list. Without this, L0's "recall memory" has nothing to recall and the system cannot compound.

L5 · Measure loop missing todayobservability layer

Nothing measures whether any of this helps. Sentry/Datadog watch the app, not the agents. The "40-60% token savings" claim (#15) is unfalsifiable as written.

Minimum viable: append one JSON line per completed task — task id, risk score, engineers spawned, review verdict, rework loops, tokens, wall-clock, escalated?. Ten runs is enough to answer the only question that matters: is Best-of-N ever winning, and is the risk threshold right? Tune the weights in agent_orchestration.config.yaml from data instead of intuition.

7 · Crowds vs experts

The governing principle: ensembles reduce variance, not bias. They pay off when errors are independent and the task decomposes. They fail when every member shares the same wrong premise, or when the work needs integrated, high-bandwidth reasoning that cannot be averaged into existence.

StageDecomposable?Independent errors?Cheap objective arbiter?Verdict
L0 Intent / specNo — integratedNo — all share the ambiguityNo Expert
L1 ArchitectureNo — high-bandwidthNoNo Expert (panel only if space is wide)
L2 ImplementationPartlyNo — correlated on the specYes — tests 1 engineer; Best-of-N only if tests arbitrate
L3 ReviewYes — dimensionsYesPartly Crowd
Search / discoveryYes — modalitiesYesNo Crowd
SynthesisNo—NoExpert

The concrete policy change

Today swarm-orchestration/SKILL.md triggers Best-of-3 at risk ≥ 0.85 and Best-of-5 at "very high" — with "only if verification can cheaply arbitrate" as an aside on the last line. That aside is the actual load-bearing condition and should be the gate.

BEFORE:  risk ≥ 0.85                        → Best-of-3
AFTER:   risk ≥ 0.85 AND objective arbiter  → Best-of-N   (tests decide; N ≤ 3)
         risk ≥ 0.85 AND NO arbiter         → narrow scope, sharpen spec, 1 engineer
                                              (an ensemble cannot arbitrate itself)

This matters more as models improve: gains from best-of-N and verifier-based test-time scaling shrink as the generator gets stronger, and coordination shows diminishing or negative returns once single-agent accuracy is high. With Opus-class engineers and no test to arbitrate, Best-of-3 mostly buys correlated noise at 3× cost — and then asks an LLM judge to pick, which reintroduces exactly the bias the ensemble was supposed to remove.

Where the crowd is genuinely underused: discovery. Finding all call sites, all affected configs, all docs that mention a renamed thing — different search modalities, blind to each other, independent errors. That is a textbook ensemble and the repo doesn't use one. (Every bug in §2 was found this way.)
